Base Teórica

Modelo Cliente-Servidor


El lado cliente corresponde al navegador web.
El lado servidor está formado por el servidor web Apache con el módulo PHP instalado y la base de datos MySQL que forman el entorno LAMP

Entorno LAMP


Un entorno LAMP está formado por un conjunto de distintas aplicaciones que forman el escenario para poder proporcionar servicios web.

  • Linux
    Sistema operativo de software libre. Distribución Ubuntu orientada al usuario medio. Gestor de paquetes que facilitan la instalación de distintas aplicaciones.
  • Apache
    Servidor web de código abierto. Atiende las peticiones HTTP entre el cliente y el servidor.
  • MySQL
    Base de datos relacional de software libre. Relaciones entre tablas. Muy rápida en operaciones de lectura.
  • PHP
    Perl o Python. El servidor web interpreta el código y genera una página web con el resultado.

Apache+PHP+MySQL

Estático vs Dinámico

Las páginas web estáticas siempre devuelven el mismo resultado HTML. Las páginas web dinámicas devuelven una respuesta HTML en función de unos parámetros recibidos desde cliente o variables dentro del servidor.

El contenido dinámico es generado por el script PHP apoyándose en la base de datos para leer y almacenar la información a representar. Tras terminar la interpretación del script PHP el servidor Apache devuelve el resultado al cliente en forma de página HTML.

PHP

Sesiones y Cookies

Son utilizadas para almacenar información.

Las cookies se guardan en el cliente, durante un periodo de vida largo, independientemente del servidor que le proporcione los recursos.

Las variables de sesión se guardan en el servidor, dependen de la conexión cliente-servidor y Pueden guargar mas información que las cookies.

La información almacenada en las cookies puede ser manipulada manualmente por el cliente, haciendo más seguras las variables de sesión. Para almacenar estructuras complejas de datos primero tendremos que serializar la información a modo de cadena.

Sintaxis

Cuestiones

Estructura

Cada cuestión tiene la siguiente estructura:

  • Enunciado.
  • Imagen principal - opcional
  • Imagen secundaria - opcional
  • Respuestas.

El enunciado de la cuestión recogerá el texto de la pregunta, pudiendo contener expresiones matemáticas representadas mediantes sintaxis LaTeX. Las imágenes tendrán un identificador secundario dentro de la cuestión. Cada respuesta tendra un valor porcentual dentro de la cuestión, siendo obligatorio que al menos una de ellas sea marcada como respuesta correcta.

Cuestiones

Expresiones matemáticas

No utilizamos imágenes creadas en otra aplicación. Utilizamos MathJax. MathJax es una librería javascript que reemplaza la expresión matemática escrita en LaTeX por las imágenes necesarias. Para que la librería pueda identificar que sustituir tenemos que rodear la expresión por un símbolo dolar $ si deseamos que este dentro del texto o por dos símbolos dolar $$ si queremos que aparezca en un párrafo a parte. Por ejemplo,

 
$$\sqrt{3x-1}+(1+x)^2$$
$$\sqrt{3x-1}+(1+x)^2$$
También podemos introducir la expresión dentro del párrafo, por ejemplo $\alpha = \omega^2 + 1$, pudiendo añadir así respuestas con expresiones matemáticas. Todo el enunciado, incluyendo la expresión matemática se guardará en un único campo.

Retos planteados

Representación y almacenamiento de la información en codificación UTF-8
fa-check y adaptando la representación a un formato exportable como PDF. También se ha eliminado la restricción de poder acceder a la aplicación desde una red exterior al servidor, en la cual se ha instalado pudiéndose acceder a ella desde cualquier navegador si el servidor en el que hemos instalado la aplicación tiene una dirección pública. Por otro lado, la aplicación no presenta la restricción de utilizarse para una única asignatura. Desde la propia aplicación gestionaremos tantas asignaturas como sean necesarias pudiendo crear cuestiones que podamos compartir y exámenes para cada una de ellas. Una de las mejoras que se puede desarrollar a partir del proyecto es modificarlo de nuevo utilizando un “framework” como Symfony2, Cake o Zend. No se utilizaron inicialmente, al requerir un conocimiento de las particularidades de estos para realizar todo el modelado de la aplicación. Las ventajas que nos ofrecerían sería más a la hora de estructurar el código para su posterior utilización en otras aplicaciones, que funcionales.
